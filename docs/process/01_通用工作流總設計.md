# 個人專案 AI 多代理開發閉環：通用工作流總設計 v1

## 1. 適用範圍

本文件適用於以下三類個人專案：

- 作品集網站：React + Vite + TypeScript + Tailwind
- 汽車官網：React + Express + styled-components
- 喝水 App：React Native

本版工作流以 **作品集網站** 作為第一個驗證樣板，並採用以下前提：

- Agent 主架構：**Hermes 為主，OpenClaw 為輔**
- 本地執行優先，不依賴 Slack / Telegram 等外部通道
- 模型以 **免費模型** 為主
- 雲端部署以 **免費方案** 為優先
- 開發環境以 **VSCode + Copilot** 為主，不依賴 Claude Code

> 目前公開資訊顯示，Hermes Agent 為開源 agent，主打長期記憶、技能生成與 learning loop；OpenClaw 則偏向 self-hosted gateway 與工作流編排，Lobster 是其 workflow shell。這個角色分工與本文件採用的架構一致。 citeturn828048search0turn828048search4turn828048search5turn828048search13

## 2. 設計原則

### 2.1 核心目標

此工作流的目標不是追求全自動，而是建立一套可重複使用的「人機協作閉環」：

1. 任務可被標準化描述
2. Agent 可依規則拆解與執行
3. 開發過程可驗證
4. 成果與經驗可沉澱
5. 下一輪任務可繼承規則與知識

### 2.2 工作流原則

- **先規格，後實作**
- **一任務一分支**
- **一代理一工作目錄**
- **測試先於合併**
- **規則必須分層管理**
- **新技術不得直接進入全域規則**
- **免費方案優先，但保留升級路徑**

## 3. 推薦角色分工

### 3.1 Director（總監代理）

負責：
- 接收需求
- 任務分類
- 判斷是否需要規格
- 指派給 PM / Dev / Test / R&D 流程
- 在驗收時對照規格進行審核

### 3.2 PM Agent

負責：
- 撰寫規格書
- 建立驗收標準
- 定義非目標（Non-goals）
- 建立測試案例草稿
- 任務狀態回寫 Notion

### 3.3 Dev Agent

負責：
- 依規格實作
- 在獨立 branch / worktree 中修改程式
- 補齊變更摘要與技術註記

### 3.4 Test / Review Agent

負責：
- 執行 lint / type check / test / build
- 檢查規格符合度
- 報告失敗原因
- 建議是否進入人工審核

### 3.5 Memory / Rules Agent

負責：
- 抽取可重複使用的規則
- 分類為 Project Rules 或 Global Rules
- 回寫 Notion 規則庫

### 3.6 R&D Agent

負責：
- 週期性搜尋新工具、新寫法與可替代方案
- 形成候選技術報告
- 不得直接改寫既有全域規則

## 4. 任務狀態機

所有任務都應使用明確狀態，不允許只以聊天訊息追蹤。

建議狀態如下：

- Inbox
- Triaged
- Spec Draft
- Ready for Dev
- In Progress
- In Review
- Test Failed
- Human Review Required
- Approved
- Deployed
- Learned

### 4.1 狀態轉移原則

- 未完成規格的 feature / refactor 不得直接進入開發
- 同一任務最多允許 agent 自修 2 輪
- 第 3 次仍失敗時，轉入 `Human Review Required`
- 僅在 `Approved` 後才允許部署
- 僅在部署或合併完成後，才能進入 `Learned`

## 5. 通用 8 步閉環

## Step 1. 任務進件（Intake）

輸入來源可為：
- 你在 VSCode / 任務清單中輸入需求
- 你在 Notion 新建任務
- 你手動提供一段功能描述

Director 先做分類：
- feature
- bug
- refactor
- content
- research
- ops

## Step 2. 任務分流（Triage）

依任務類型決定後續流程：

- `content`：可走簡化流程
- `bug`：需先建立重現條件
- `feature`：必須建立規格與驗收標準
- `refactor`：必須列出不可破壞範圍
- `research`：只進入候選知識，不直接改規則

## Step 3. 規格生成（Spec）

PM Agent 建立規格書，至少包含：
- Goal
- Scope
- Non-goals
- Technical Constraints
- Acceptance Criteria
- Test Cases
- Risks
- Done Definition

## Step 4. Worktree 建立與任務交付

每個任務都必須在 **獨立 branch + 獨立 git worktree** 執行。

建議結構：

```bash
repo/
worktrees/
  feature-home-hero/
  bug-contact-form/
  refactor-navbar/
```

建立方式：

```bash
git worktree add -b feat/home-hero ../worktrees/feature-home-hero
```

Git 官方文件明確指出，同一個 repository 可支援多個 working tree，讓多個分支同時 checkout 與並行開發；這正是多代理隔離的基礎。 citeturn426605search1

## Step 5. 實作與自我檢查

Dev Agent 或人工開發需完成：
- 實作功能
- 補必要註解
- 自查是否符合規格
- 紀錄變更影響範圍

## Step 6. 自動驗證

至少執行：
- lint
- type check
- unit test
- build

依專案補充：
- 作品集：Lighthouse / basic accessibility check
- 汽車官網：前後端整合檢查、API env 檢查
- React Native：navigation / local storage / reminder 行為檢查

## Step 7. 規格對照審核

Director / Review Agent 以規格為準，檢查：
- 是否符合 Acceptance Criteria
- 是否引入 scope creep
- 是否遺漏測試案例
- 是否需要人工介入

## Step 8. 知識沉澱與規則更新

通過後，將成果拆成三類：
- Task Log
- Bug Fix Pattern
- Reusable Rule

規則分成：
- Global Rules
- Project Rules
- Candidate Rules（待觀察）

## 6. Git Worktree 納入正式規範

你提供的文章觀念正確，而且非常適合多代理流程，因此本文件將其升級為**強制規則**。

### 6.1 使用 Worktree 的原因

若多個 Agent 在同一工作目錄中操作，容易出現：
- 覆蓋同一檔案
- 切 branch 時污染工作區
- stash 混亂
- node_modules / build cache 相互影響
- 無法追蹤哪個變更是由哪個 Agent 完成

### 6.2 Worktree 管理規範

- 每一個 active task 必須對應一個 worktree
- 每一個 worktree 只能對應一個 branch
- 不允許兩個 Agent 共用同一 worktree
- 任務完成後，merge / close 後應移除 worktree
- worktree 路徑應統一放在 `/worktrees` 或 repo 同層資料夾

### 6.3 命名規則

建議格式：

- `feature-<task-slug>`
- `bug-<task-slug>`
- `refactor-<task-slug>`
- `research-<task-slug>`

例如：

- `feature-home-hero`
- `bug-contact-email-validation`
- `refactor-project-card-layout`

### 6.4 Worktree 生命週期

1. Director 核准任務
2. 建立 branch
3. 建立 worktree
4. 開發與測試
5. 提交 PR / merge
6. 移除 worktree
7. 記錄 learnings

## 7. 免費模型與免費雲端前提下的架構建議

### 7.1 模型策略

你已明確限制：
- 只用免費模型
- 不依賴付費 Claude 類服務

因此流程設計必須偏向：
- 結構化提示詞
- 小步驟任務拆解
- 減少 agent 重規劃次數
- 使用 Lobster 類 workflow shell 提高 determinism
- 使用 Hermes 做長期記憶，不讓每輪都從零開始

OpenClaw/Lobster 官方公開描述其價值在於把多步驟 tool sequence 包成較具確定性的 workflow，減少反覆 re-planning。這很適合免費模型場景。 citeturn828048search1turn828048search5turn828048search17

### 7.2 免費雲端策略

建議如下：

- **前端**：Netlify Free
- **資料庫 / Auth / Storage**：Supabase Free
- **輕量後端 API**：Cloud Run 免費額度內，或以 Supabase Edge Functions 優先

目前公開方案顯示：
- Netlify 提供 always-free free plan，且公開說明 free plan 有硬性月度上限，不會自動超支。 citeturn828048search2turn828048search14
- Supabase 文件顯示 Free Plan 提供 2 個 free projects。 citeturn828048search7turn828048search19
- Cloud Run 官方說明仍有 free tier，且每月重置，超出後才計費。 citeturn426605search0turn426605search2

### 7.3 免費方案下的限制意識

必須把以下項目寫入工作流檢查：
- 大圖與影片不得直接塞進 Netlify build output
- Supabase Storage 僅放必要媒體資產
- 後端服務預設 scale-to-zero
- 避免重型 cron / 常駐 worker
- 不做高頻背景任務
- 優先靜態化與快取

## 8. 架構建議：Hermes 為主，OpenClaw 為輔

### 8.1 Hermes 適合擔任的部分

- 長期記憶
- 規格模板累積
- 規則演化
- 過往任務搜尋
- 專案風格偏好學習
- R&D 結果整理

Hermes 官方文件與 GitHub README 將其定位為具 built-in learning loop 的 agent，並強調可從過往經驗中生成技能與持久化知識。 citeturn828048search0turn828048search4turn828048search8turn828048search12

### 8.2 OpenClaw / Lobster 適合擔任的部分

- 本地入口與通路整合
- workflow 編排
- approval checkpoints
- 固定操作流程封裝
- 工具呼叫與外部任務串接

OpenClaw 官方將自己定位為 self-hosted gateway；Lobster 則是 workflow shell，適合將標準作業封裝成 deterministic operation。 citeturn828048search5turn828048search9turn828048search13

### 8.3 實際建議

你的現況最適合的做法是：

- **Hermes**：負責記憶、規格、規則、R&D
- **OpenClaw/Lobster**：負責觸發與執行流程
- **VSCode + Copilot**：負責日常人工開發與修改

這比「完全交給單一 agent」更穩定，也較符合免費模型的能力上限。

## 9. 作品集網站作為第一個驗證樣板

作品集專案的目標不宜過重，建議先用來驗證：

- 規格模板是否好用
- Worktree 是否順手
- Notion 任務流是否清楚
- Global Rules 與 Project Rules 是否能分層
- 測試與 build check 是否能閉環

### 9.1 作品集優先驗證任務建議

建議從以下類型開始：

1. Hero 區塊重構
2. 專案卡片元件統一
3. 響應式 navbar 調整
4. 聯絡表單驗證
5. SEO meta / OG image 補齊
6. Lighthouse 優化

這些任務夠小、風險低、又能測出 workflow 是否順暢。

## 10. 導入順序

### 階段 1：文件先行

先建立：
- 通用工作流設計
- PM 規格模板
- Notion schema
- Global Rules

### 階段 2：作品集實測

對作品集執行 3–5 個小任務：
- 每個任務一個規格
- 每個任務一個 worktree
- 每個任務一次驗收

### 階段 3：規則沉澱

從作品集中抽出：
- 前端命名慣例
- 元件切分規則
- Tailwind / TypeScript 習慣
- commit 與 PR 模板

### 階段 4：移植到汽車官網

再補上：
- Express API 規範
- env 管理
- 部署檢查
- Supabase / Cloud Run 邊界

## 11. 汽車官網專案的額外規範

由於你要求把汽車官網部署規則納入通用設計，因此增加以下限制：

### 11.1 部署責任分層

- 前端：Netlify
- 資料與認證：Supabase
- 後端 API：Cloud Run 或其他 free-tier 服務

### 11.2 部署前檢查

- `.env.example` 是否完整
- 前端 API base URL 是否可切換環境
- CORS 設定是否與 Netlify 網域對齊
- Supabase keys 是否分清 public / service role
- Cloud Run 是否關閉不必要常駐行為
- production build 是否可在無本地秘密金鑰情況下完成

### 11.3 後端設計約束

- 不設計重型常駐 worker
- 優先 stateless API
- 優先使用 managed service
- 能用資料庫 function / edge function 解決者，不額外拉一個常駐 backend

---

## 12. 最低可行落地版本（MVP Workflow）

若你要最先落地，可先只做以下內容：

1. Notion 建 4 個資料庫
2. 建立 PM Spec 模板
3. 建立 Global Rules 頁面
4. 每任務使用一個 git worktree
5. 作品集先跑 3 個 feature / bug 任務
6. 每次任務結束都寫一次 learnings

做到這一步，你就已經不是「討論工作流」，而是開始真正擁有可複用的開發系統。
